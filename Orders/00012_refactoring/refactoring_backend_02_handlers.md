# ガイドライン: バックエンド ハンドラー処理 リファクタリング

対象の担当者: Backend-Handler-Agent

## 目的

バックエンドのエンドポイントハンドラー内のエラーハンドリング手法の一元化・統一を行います。
他のエージェントと並行稼働するため、コンフリクトを避けるべく指定された【対象ファイル】以外は極力変更しないでください。

## 対象ファイル

- `backend/handlers/*.go` (特に `todo.go`, `auth.go` 等)
- `backend/handlers/*_test.go`
  （抽出先として `backend/utils/` などに関連する新規ファイルを作成することは可能です）

## やるべきこと

1. **エラーハンドリングの共通化**:
   - 現在、`todo.go` をはじめとするハンドラーの各メソッド内に `errorHandling` というクロージャー関数が冗長的に定義されています。
   - これを各メソッド内から削除し、ハンドラーパッケージの外部（またはEchoのカスタムエラーハンドラ `e.HTTPErrorHandler` 等）へ共通ヘルパー関数として抽出し、一元化・クリーンアップしてください。
2. **エラーログの一貫性確保**:
   - エラーをレスポンスとして返す際、一部のメソッドでしかログやスタック情報が出力されていません。`slog` 等を用いたログ機能で統一的にエラーハンドリングできるようにしてください。
3. **トレーサビリティの確立**:
   - リクエストを処理する先頭でリクエスト内容をログに出力するようにしてください。
   - エラーが発生した場合は、エラー内容をログに出力するようにしてください。
4. **テストの修正と通過（必須要件）**:
   - エラーハンドリングの変更により仕様が変わるテストケースがある場合は、適宜修正してください。
   - 実装作業後、必ず `backend/` ディレクトリに移動して `go test ./...` を実行し、**すべてのテストがパスすることを確認**してから作業を完了してください。
   - **注意: 絶対に `git commit` や `git add` 等のgit操作は実行しないでください。** 修正のコミットや `pre-commit` フックの実行はすべてのエージェントの作業完了後にメインエージェントがまとめて行います。コードの修正とテスト確認のみを行って終了してください。
