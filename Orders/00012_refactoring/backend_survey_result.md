# Backend Refactoring Survey Result

このドキュメントは、`backend/` ディレクトリ配下のコードベースを調査し、リファクタリングの余地がある箇所をまとめたものです。

## 1. アーキテクチャと依存性注入 (Google Wire)

- **Service Factory の廃止と直接注入**:
    - `TodoHandler` で `TodoServiceFactory` を使用してリクエストごとにサービスを生成していますが、これはボイラープレートを増やしています。`TodoService` を直接 `TodoHandler` に注入するように変更すべきです。
- **リポジトリへの依存性注入**:
    - 現在、リポジトリやサービス内で `ent.FromContext(ctx)` を使用して DB クライアントを取得していますが、`ent.Client` をリポジトリの構造体に直接保持（DI）する方が一般的で、テストも容易になります。
- **ロガーの扱い**:
    - `slog.Logger` も同様に、各レイヤーの構造体に DI されるべきです。

## 2. エラーハンドリング

- **ハンドラー内での共通エラー処理の重複**:
    - `handlers/todo.go` の各メソッド内で `errorHandling` クロージャが定義されており、コードが冗長です。Echo のカスタムエラーハンドラー（`e.HTTPErrorHandler`）を利用するか、共通のヘルパー関数に抽出して一元化すべきです。
- **エラーログの不整合**:
    - メソッドによってエラーログを出力する場合としない場合があり、一貫性がありません。

## 3. コンテキスト管理

- **コンテキストキーの型安全性**:
    - `context.WithValue(ctx, "user", user)` のように文字列をキーとして使用しています。これは衝突のリスクがあるため、Go のベストプラクティスに従い、エクスポートしない独自型（`type contextKey string` 等）をキーとして使用すべきです。
- **Middleware との連携**:
    - ユーザー情報のコンテキストへのセットと、リポジトリでの取得ロジックをより型安全かつ明示的にすべきです。

## 4. 設定と環境変数

- **ハードコードされた値**:
    - `providers/ent_provider.go` の DB 接続文字列
    - `routes/router.go` の CORS 許可オリジン
    - `middleware/auth.go` の JWT シークレットのフォールバック値
    - `server.go` のポート番号 (`:8080`)
    - これらはすべて環境変数から取得するようにリファクタリングすべきです。

## 5. バリデーション (go-playground/validator)

- **エラーメッセージのマッピングの共通化**:
    - `validators/*.go` 内で、各リクエスト構造体ごとにバリデータタグを日本語メッセージに変換する巨大な switch 文があります。これは非常に冗長です。`go-playground/validator` の `Universal Translator` 等を使用して、グローバルにメッセージを管理すべきです。

## 6. ミドルウェア

- **トークンのリフレッシュロジック**:
    - `AuthMiddleware` 内でリクエストごとに新しい JWT トークンを生成して Cookie を更新していますが、これは一般的ではなく、パフォーマンスやセキュリティ設計（リフレッシュトークンの分離など）の観点で見直すべきです。
- **CSP Nonce の固定値**:
    - `router.go` で CSP の nonce が `'nonce-random123'` と固定されています。これはセキュリティ上の意味をなさないため、リクエストごとに動的に生成すべきです。

## 7. テスト

- **単体テストの改善**:
    - `TodoService` のテストに `spyTodoRepo` を手動で実装していますが、`mockery` などのツールを使用してモックを自動生成することで、メンテナンス性を向上させることができます。
- **結合テストのボイラープレート**:
    - 各テストでの Echo のセットアップ、DB クリーンアップ、認証リクエストの作成などが重複しています。テストヘルパーやスイートを導入して簡略化すべきです。
- **トランザクションのテスト**:
    - `utils.WithTx` を使用している箇所のテストが、依然として実 DB（SQLite 含む）に依存しています。トランザクション管理自体を抽象化することで、より純粋な単体テストが可能になります。

## 8. コードスタイルとリンター

- **DTO と Entity の分離**:
    - サービスレイヤーが `ent.Todo` (Entity) を返し、ハンドラーで `dto.EntityToTodoDto` を呼んでいる箇所があります。サービスレイヤーで DTO を返すように統一するか、境界をより明確にすべきです。
- **リンター設定の強化**:
    - `.golangci.yml` が最小限の設定しかありません。`gocyclo`, `revive`, `gocritic` などを導入して、コード品質を自動的にチェックできるようにすべきです。
