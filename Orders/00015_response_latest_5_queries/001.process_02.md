# サブエージェント向け作業手順書: 検索履歴APIのレビュー指摘事項の対応

## 概要

作成した検索履歴APIの実装に対し、`001.review.md` に記載された指摘事項への対応を行います。
各フェーズ（指摘事項1点ごと）の作業が完了したら、全テストが通ることを確認し、それぞれ単独のコミットを作成してください。

## フェーズ1: ユーザー取得処理のリポジトリ層への隠蔽

1. **対象ファイル**
   - `backend/repositories/todo_filter_history.go`
   - `backend/services/todo_filter_history.go`
   - `backend/handlers/todo.go`
   - `backend/testutils/mock_todo_filter_history_repo.go`
   - および関連するテストコード (`backend/services/todo_filter_history_test.go`, `backend/handlers/todo_test.go`)

2. **実装内容**
   - `backend/repositories/todo_filter_history.go`: `TodoRepository` の実装を参考に、コンテキストからユーザーを取得する関数等を用意するか直接取得し、`FetchLatestFilters` メソッドからは引数 `userID int` を削除してください。対象のインターフェース `ITodoFilterHistoryRepository` も忘れず修正してください。
   - `backend/services/todo_filter_history.go`: リポジトリの仕様変更に伴い、`FetchLatestFilters` メソッドから引数 `userID int` を削除し、インターフェース `ITodoFilterHistoryService` も修正してください。
   - `backend/handlers/todo.go`: `ListTodoFilterHistories` メソッドにおいて、`context` からユーザー情報を手動で取り出す処理（`ctx.Value("user")` 等）や `userID` をサービスへ渡す処理を削除し、サービス呼び出しを `FetchLatestFilters(ctx)` とのみするように修正してください。

3. **テスト実装の修正**
   - `backend/testutils/mock_todo_filter_history_repo.go` のモックメソッドのシグネチャと `m.Called` の引数を修正してください。
   - `backend/services/todo_filter_history_test.go` にて、サービスに対するテストケースの引数を修正し、モック呼び出し部分も合わせて修正してください。
   - `backend/handlers/todo_test.go` でビルドエラーが出る場合は適切に修正してください。

4. **完了条件**
   - [ ] `go test ./...` ですべてのテストが成功すること。
   - [ ] 「ユーザー情報の取得元をリポジトリ層でコンテキストから取得するように修正」の旨が分かる明確なメッセージでコミットを作成すること（このフェーズの変更内容のみを含めること）。

## フェーズ2: レスポンス用構造体の DTO 化

1. **対象ファイル**
   - `backend/dto/todo.go`
   - `backend/handlers/todo.go`

2. **実装内容**
   - `backend/handlers/todo.go` の `ListTodoFilterHistories` 内のローカルで定義されている匿名構造体（`queries` 配列の型など）を `backend/dto/todo.go` に移動させてください。
   - 型名は `ListTodoFilterHistoriesResponseDto` および `TodoFilterHistoryQueryDto` （名称はプロジェクトの規則に合わせて適宜調整）のようにわかりやすく定義してください。
   - `backend/handlers/todo.go` 側では該当の DTO を参照してレスポンス JSON を組み立てるよう修正してください。

3. **テスト実装の修正**
   - `backend/handlers/todo_test.go` などのテストで該当構造体を利用して Unmarshal している部分はそのままローカルな構造体定義を利用して検証しても良いですが、もし DTO に依存している箇所があれば適切にリファクタリングしてください。

4. **完了条件**
   - [ ] `go test ./...` ですべてのテストが成功すること。
   - [ ] 「検索履歴レスポンスの構造体を DTO に分離」の旨が分かる明確なメッセージでコミットを作成すること（このフェーズの変更内容のみを含めること）。
