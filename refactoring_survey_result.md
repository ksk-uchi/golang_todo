# golang_todo リポジトリ リファクタリング調査報告書

本ドキュメントは、`gemini-cli` サブエージェントを用いて `golang_todo` リポジトリの `backend` および `frontend` それぞれのコードベースを調査し、リファクタリングの余地がある箇所をまとめた最終成果物です。

## Backend (Go/Echo) のリファクタリング課題

### 1. アーキテクチャと依存性注入 (Google Wire)

- **Service Factory の廃止と直接注入**:
  - `TodoHandler` で `TodoServiceFactory` を使用してリクエストごとにサービスを生成していますがボイラープレートを増やしています。`TodoService` を直接 `TodoHandler` に注入するように変更すべきです。
- **リポジトリへの依存性注入**:
  - 中で `ent.FromContext(ctx)` を使用してDBクライアントを取得していますが、`ent.Client` をリポジトリの構造体に直接保持（DI）する方が一般的でテストも容易になります。
- **ロガーの扱い**:
  - `slog.Logger` も同様に各レイヤーの構造体にDIされるべきです。

### 2. エラーハンドリング

- **ハンドラー内での共通エラー処理の重複**:
  - `handlers/todo.go` の各メソッド内で `errorHandling` クロージャが定義されており冗長です。Echo のカスタムエラーハンドラーを利用するか、共通のヘルパー関数に抽出して一元化すべきです。
- **エラーログの不整合**:
  - メソッドによってエラーログを出力する場合としない場合があり一貫性がありません。

### 3. コンテキスト管理

- **コンテキストキーの型安全性**:
  - `context.WithValue(ctx, "user", user)` のように文字列をキーとして使用しており衝突のリスクがあります。非公開の独自型をキーとして使用すべきです。
- **Middleware との連携**:
  - ユーザー情報のコンテキストへのセットと取得ロジックをより型安全かつ明示的にすべきです。

### 4. 設定と環境変数

- **ハードコードされた値の排除**:
  - DB 接続文字列、CORS 許可オリジン、JWT シークレットのフォールバック値、ポート番号 (`:8080`) などがハードコードされており、環境変数から取得するようリファクタリングすべきです。

### 5. バリデーション (go-playground/validator)

- **エラーメッセージのマッピングの共通化**:
  - 各リクエスト構造体ごとにバリデータタグを日本語メッセージに変換する巨大な switch 文があり非常に冗長です。Universal Translator 等を使用してグローバルにメッセージを管理すべきです。

### 6. ミドルウェア

- **トークンのリフレッシュロジック**:
  - AuthMiddleware 内でリクエストごとに新しい JWT トークンを生成して Cookie を更新していますがパフォーマンスやセキュリティ設計の観点で見直すべきです。
- **CSP Nonce の固定値**:
  - CSP の nonce が固定値（`'nonce-random123'`）になっています。リクエストごとに動的に生成すべきです。

### 7. テスト

- **単体テストと結合テストの改善**:
  - `mockery` 等を使用してモックを自動生成することでメンテナンス性を向上させ、テスト時の Echo のセットアップ等のボイラープレートをテストヘルパーで簡略化すべきです。

### 8. コードスタイルとリンター

- **DTO と Entity の分離**:
  - サービスレイヤーからの戻り値（Entity か DTO か）の境界をより明確に統一すべきです。
- **リンター設定の強化**:
  - `.golangci.yml` に `gocyclo`, `revive`, `gocritic` などを導入してコード品質チェックを自動化すべきです。

## Frontend (Next.js/TypeScript) のリファクタリング課題

### 1. コンポーネントの肥大化とロジックの混在

- **`src/app/page.tsx` (`HomeContent`)**:
  - 状態管理、API通信、トースト通知、ページネーションなどが一箇所（約200行）に集中しています。カスタムフック (`useTodos` 等) に状態とロジックを分離すべきです。
- **`TodoModal.tsx` / `LoginPage.tsx`**:
  - フォームの状態管理とバリデーションが手動で行われています。

### 2. API通信・通信基盤の課題

- **ハードコードと直接通信**:
  - `baseURL` がハードコードされており環境変数を利用すべきです。
  - 各コンポーネントで `axios` を直接呼び出しているため、Service層等を作成して抽象化すべきです。
  - CSRFトークンの抽出が不安定な実装です。

### 3. バリデーションと型定義

- **Zodの未使用**:
  - インストールされている `zod` が活用されていません。フォームバリデーションに導入すべきです。
- **型安全性の向上**:
  - `as Error` や型アサーション (`error as { ... }`) が見られ、適切に型ガードを導入すべきです。

### 4. ディレクトリ構成とコンポーネントの分類

- **共通UIとドメイン固有コンポーネントの混在**:
  - `src/app/components` に純粋なUI部品とドメイン部品（`TodoItem`等）が混ざっています。`components/ui` と `components/todo` のように整理が推奨されます。
- **UI/UXの一貫性**:
  - トースト通知の設定（`toastHandler`）が一部コンポーネント内に閉じているため、グローバルな設定を使用すべきです。

### 5. その他

- **`suppressHydrationWarning`**:
  - 警告を無闇に抑制している可能性があり、根本原因の解消を検討すべきです。
- ディレクトリ固有ルール（`pnpm`等）に関しては既存の通り徹底されています。
